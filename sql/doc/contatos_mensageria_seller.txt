
DOCUMENTAÇÃO ESTRUTURAL — BASE MENSAGERIA SELLER
================================================

5.0 RESUMO DO CONTEXTO DA BASE
------------------------------
Domínio declarado:
Interações entre clientes e sellers 3P em sistema de mensageria do marketplace.

Finalidade estrutural:
Padronizar registros de mensageria cliente–seller para um formato compatível
com modelo de tickets de atendimento.

Fluxo operacional descrito:
Cliente entra em contato com o seller via mensageria.
Essas interações não passam diretamente pelo atendimento humano.
Quando ocorre mediação, pode ser gerado ticket na plataforma de atendimento.

Escopo da documentação:
Documentar estrutura, relações, granularidade aparente, chaves operacionais
e limites de entendimento da query fornecida.

Origem do contexto:
- Contexto humano fornecido
- Query SQL
- Amostra de dados

Limitações:
- Ausência de dicionário de dados oficial
- Estrutura completa das tabelas de origem não fornecida
- Amostra limitada de registros

-----------------------

-------------------

Relação 1

Relação 2
pedped.id_pedped = pedprd.id_pedped

Dependência estrutural observada:

sessions
   └── pedped
         └── pedprd

5.5 HIPÓTESES EXPLÍCITAS
------------------------

A granularidade final da query pode ser impactada pelo
join com itens do pedido.

Caso um pedido possua múltiplos SKUs, múltiplas linhas
podem ser geradas para um mesmo contato.


5.6 RISCOS E PONTOS DE ATENÇÃO
------------------------------

Possível fan-out estrutural devido ao join entre
pedido e itens do pedido.

Multiplicação de registros pode ocorrer quando
um pedido possui mais de um SKU.


5.7 LIMITES DO ENTENDIMENTO
---------------------------

Não é possível afirmar:

- Quantas sessões podem existir por pedido
- Se existem múltiplas mensagens dentro da sessão
- Estrutura completa das tabelas de origem
- Se a sessão representa uma thread completa de conversa

7.0 MAPA DE GRANULARIDADE
-------------------------

Níveis estruturais observados:

Pedido Pai
Agrupador de pedidos

Pedido
Compra individual do cliente

Contato
Contato do cliente associado ao pedido

SKU
Item ou produto do pedido


Hierarquia estrutural:

Pedido Pai
   │
   └── Pedido
          │
          └── SKU


Granularidade aparente da saída:

Contato do cliente
        x
SKU do pedido


8.0 MAPA DE FAN-OUT
-------------------

Estrutura de relacionamento:

sessions
   │
   │ (associação com pedido)
   ▼
pedido
   │
   │ 1 : N
   ▼
SKU do pedido


Exemplo estrutural hipotético:

1 sessão
   │
   ▼
1 pedido
   │
   ├── SKU 1 → linha 1
   ├── SKU 2 → linha 2
   └── SKU 3 → linha 3


Nesse cenário, um único contato pode gerar
múltiplas linhas na saída da query caso o
pedido possua múltiplos SKUs.


8.0 QUERY


SELECT
    CAST(s."order" AS INT) AS id_contato,
    s.created_at AS data_atendimento,
    CAST(fc.id_pedped AS INT) AS id_pedido,
    CAST(fc.parent_pedped AS INT) AS id_pedido_pai,
    CAST(prd.id_prdprd AS INT) AS id_produto,
    'Nova Mensageria' AS motivo_contato,
    'Nova Mensageria' AS submotivo_contato,
    'Nova Mensageria' AS email_atendente,
    'Nova Mensageria' AS email_solicitante,
    'Nova Mensageria' AS email_atribuido,
    'Nova Mensageria' AS group_name,
    'Nova Mensageria' AS origem_atendimento,
    'Mensageria Seller' AS origem_informacao,
    'Nova Mensageria' AS flag_reputacional,
    'Nova Mensageria' AS motivo_contato_voz,
    'Nova Mensageria' AS canal_atendimento,
    'Nova Mensageria' AS tipo_ticket,
    'Nova Mensageria' AS status_ticket,
    'Nova Mensageria' AS descricao_ticket,
    'Nova Mensageria' AS flag_tags,
    'Nova Mensageria' AS formulario,
    'Nova Mensageria' AS resolucao_cx
FROM lake_mensageria_seller.sessions AS s
    INNER JOIN lake_madeira_fc.b2c_pedped AS fc ON fc.id_pedped = s."order"
    INNER JOIN lake_madeira_fc.b2c_pedprd AS prd ON fc.id_pedped = prd.id_pedped
WHERE s.created_at >= '2025-04-01'
    AND s."source" NOT IN ('marketplace')
    AND s.seller_id NOT IN (1,8)
